# 引き継ぎ・移行プラン

**最終更新**: <!-- AUTO:last-updated -->
2026-07-08
<!-- /AUTO:last-updated --> / **対象読者**: 北林 / CTO 候補 / 将来雇用するエンジニア

ReliefNote MVP は Vanilla JS + Cloudflare Workers + Supabase で構築されています。
将来 B2B 契約獲得後に技術責任者を迎えるとき、どのレベルの再実装を選ぶかで投資効率が大きく変わります。
本書は **3 つの選択肢** を比較し、それぞれのメリット/デメリット/移行コストを示します。

---

## 全体像: 何が "資産" として残るか

実装言語に関わらず、以下の資産は全パターンで継承可能:

| 資産 | 言語非依存性 | 価値 |
|---|---|---|
| **CSV マスター 9 本** (制度知識) | ★★★ 完全に独立 | 事業の核 |
| **ルール DSL の仕様** | ★★★ 仕様書として残る | ビジネスロジック |
| **Supabase スキーマ** (SQL) | ★★★ 言語非依存 | DB 層 |
| **検証済み UX フロー** | ★★★ 設計知見 | 再実装の地図 |
| **パートナー実績・契約** | ★★★ 信頼資産 | 継続性 |

捨てになる可能性があるのは「**特定言語で書いた実装コード**」のみ。MVP 全体価値の 30% 程度。

## 選択肢 A: 現状維持 (CTO 不要、自社運用継続)

### 構成
```
[Vanilla JS Frontend]  ←変えず
[Cloudflare Workers]   ←変えず
[Supabase]             ←変えず
```

### 適するケース

- パートナー要望がコンテンツ更新中心

- 自分以外のエンジニアが入りづらい

### 月額コスト目安

| GitHub Pages | ¥0 |

| **合計** | **~¥3,000-10,000/月** |

## 選択肢 B: フロント維持 + バックエンド分離 (推奨)

### 構成
```
[Vanilla JS Frontend]  ←そのまま
       ↓ REST API
[新規バックエンド層]    ←CTO候補が作る
   - PHP/Laravel または Node/Hono
   - REST API / GraphQL
   - 管理画面 (パートナー向け)
       ↓
[Supabase Postgres or 独自DB]
[Cloudflare Workers (AI proxy)]  ←そのまま
```

### 適するケース
- B2B 契約 3〜10 件
- パートナー側に管理画面が必要
- CTO 候補が既存資産を尊重して開発に参加してくれる

### 移行ステップ
1. 既存 `index.html` の Supabase 直叩きを **fetch ラッパー化** (1 週間)
2. CTO 候補が REST API を実装 (Laravel / Hono) (4〜8 週間)
3. ラッパーの呼び先を REST API に切り替え (数日)
4. パートナー向け管理画面は新スタック (PHP Blade / React など) で別途構築 (4〜8 週間)
5. AI Worker と Cloudflare 周りはそのまま継続

### 引き継ぐ資産
- フロント実装コード ✅ (= MVP の 70%)
- CSV マスター ✅
- DB スキーマ ✅
- AI 機能 ✅

### メリット
- 北林さんの実装投資が大きく残る
- フロントとバックエンドの責務が明確化される
- CTO 候補の技術選択 (PHP/Node 等) を尊重できる
- パートナーごとに管理画面だけカスタムしやすくなる
- 段階的に移行できるためリスク低い

### デメリット
- API レイヤーを設計する手間
- 認証フロー (誰が何を読み書きできるか) を再設計する必要

### コスト目安
| 項目 | 金額 |
|---|---|
| 既存サービス継続 | ~¥3,000-10,000/月 |
| 追加サーバ (PHP/Node) | ~¥5,000-30,000/月 (Cloudflare Pages or VPS) |
| CTO 候補開発費 | 200〜400万 (12週間想定) |
| **合計 (運用)** | **~¥10,000-40,000/月** |

### 大津さんとの相性
PHP+JavaScript で完結できる構成。**最もありそうな着地点**。

## 選択肢 C: 全面書き換え (PHP / モダンフレームワーク)

### 構成
```
[全面再実装]           ←CTO候補が一から作る
   - PHP/Laravel 全部
   または
   - Next.js + Node API 全部
[新規 DB 設計]
[Cloudflare Workers (AI proxy)]  ←残せる場合は残す
```

### 適するケース
- B2B 契約 10 件超 + Series A 等の本格投資フェーズ
- 既存実装に致命的な制約 (マルチテナント・リアルタイム・etc.) がある
- 北林さんが「過去の MVP は捨てる前提」で割り切れる

### 移行ステップ
1. 要件定義 (4 週間) — 既存の振る舞いをすべて仕様書化
2. 設計 (4 週間) — DB / API / UI / 権限モデル
3. 実装 (16 週間) — フロント・バック・管理画面・テスト
4. 並行運用 (4 週間) — 新旧並行稼働で検証
5. 切り替え (2 週間) — DNS 切替・パートナー周知

### 引き継ぐ資産
- CSV マスター ✅ (DB に import すれば OK)
- ルール DSL の仕様 ✅ (再実装)
- DB スキーマ (一部) ✅
- フロント実装コード ❌ **捨てる**

### メリット
- 技術スタックを完全に CTO 主導で決められる
- マルチテナント設計を最初から織り込める
- ブランディング/カスタムも自由

### デメリット
- 開発期間 6〜9 ヶ月、開発費 500〜800 万円
- 既存パートナーへの影響大 (機能パリティ達成まで運用が二重)
- 既存の UX で得た最適解を再発見する必要

### コスト目安
| 項目 | 金額 |
|---|---|
| 既存サービス継続 (並行期間) | ~¥3,000-10,000/月 |
| 新規サーバ | ~¥10,000-50,000/月 |
| CTO + チーム開発費 | 500〜800 万 (6〜9ヶ月) |
| **合計 (運用)** | **~¥15,000-60,000/月** |

### 大津さんとの相性
大津さんの当初提案 (PHP 4ヶ月 300万) はこの選択肢に近いが、機能スコープ次第で実際は 500〜800 万になる可能性が高い。

## 推奨パス

```
[今] フェーズ1 (PoC / B2B 0〜2件)
  └─ 選択肢 A (現状維持) を継続
     - 営業に全リソース
     - ドキュメント整備のみ技術投資
       ↓ B2B 契約 3〜5 件
[+1年] フェーズ2-3 (B2B 拡大)
  └─ 選択肢 B (フロント維持 + API 分離) へ移行
     - CTO 候補と「フロント維持できますか？」を擦り合わせ
     - 大津さんが PHP API を担当する形が現実的
     - 管理画面だけ別フレームワークで新規構築
       ↓ B2B 10件超 + 投資調達
[+2-3年] フェーズ4 (スケーリング)
  └─ 選択肢 C (全面書き換え) を判断
     - マルチテナント・SaaS 化が事業要件として明確になった時点で
     - それまでに溜めたデータと UX の知見を活かして再実装
```

## CTO 候補との会話で確認すべきこと

選択肢 B (推奨) の現実性を測るため、CTO 候補に必ず以下を聞くこと:

1. **「既存の Vanilla JS フロントエンドを残したまま、バックエンド (REST API) だけを担当できますか？」**  
   → Yes ならば選択肢 B が現実的。No (= 全部書き換えたい) なら選択肢 C のコストを覚悟するか、別の CTO を探す
2. **「PHP 以外の言語 (Node.js, TypeScript, Go, Python) にも対応できますか？」**  
   → 既存 Cloudflare Worker (TypeScript) と統合する場合、Node 系の方が継承しやすい
3. **「Supabase の継続使用で問題ないですか？」**  
   → 独自 DB を立てたいと言われたら、移行計画と運用コストを再計算する必要
4. **「家族共有の E2E 暗号化スキームを維持できますか？」**  
   → Web Crypto API ベースの実装は仕様書化済み。サーバ側は暗号化済みデータの保管庫として動かせばよい
5. **「マルチテナント (パートナー別 DB 分離) はどう設計しますか？」**  
   → これは選択肢 C の領域。すぐは不要だが、設計思想を聞いておく

## まとめ

- **MVP は捨てない選択肢 B が "ほぼ必ず正解"** (実装の 70% を引き継げる)
- 大津さんが PHP 専業でも、選択肢 B なら大きな支障なく協業可能
- **全面書き換え (選択肢 C) は B2B 契約 10 件超まで保留** が経済合理的
- どの選択肢でも、**ドキュメント (この `docs/` 一式) と CSV マスターは 100% 引き継げる資産**
- 「捨てる前提」を強要する CTO 候補は、その判断自体を CTO 候補としての適性評価に使える
